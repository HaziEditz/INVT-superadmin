Dedup logic (double-count guard):

// Query completedJobs for this booking from ALL writers
firebase.database()
  .ref('completedJobs/' + cid)
  .orderByChild('bookingId').equalTo(bookingId)  // requires .indexOn ["bookingId"] in rules
  .once('value')
  .then(function(existSnap) {
    var count = 0;
    existSnap.forEach(function() { count++; });
    if (count > 1) {
      // Driver app already wrote a push-key completedJobs record for this booking
      // AND incremented earnings — we skip to prevent double-count.
      return;
    }
    // Only our record → safe to increment
    _deRef.set({
      driverName:    _cj.driverName || existing.driverName || '',
      totalEarned:   (existing.totalEarned   || 0) + fare,
      pendingAmount: (existing.pendingAmount  || 0) + fare,
      tripCount:     (existing.tripCount      || 0) + 1,
      lastPaidAt:    existing.lastPaidAt,    // preserved exactly, even if 0
      updatedAt:     Date.now(),
    });
  });

Critical Firebase rule required for the dedup query to work:

"completedJobs": {
  "$companyId": {
    ".indexOn": ["bookingId"]
  }
}

Without this index, the orderByChild('bookingId') query still works but Firebase warns and does a full table scan — fine at low volume, degrades at scale.

The lastPaidAt field is deliberately preserved using hasOwnProperty rather than falsy-check, because 0 is a valid stored value (driver has never been paid — timestamp of zero).

Root cause of §108d: before this fix, the completion path only wrote Status:'Completed' to allbookings and left paymentMethod as whatever was stored at booking-creation time — which for web bookings was whatever the Stripe webhook wrote (sometimes 'card', sometimes absent). The IIFE now always writes both cases of paymentMethod using the four-field resolution chain, server-side, using a service-account token so it fires even if the dispatcher's browser session has expired.
