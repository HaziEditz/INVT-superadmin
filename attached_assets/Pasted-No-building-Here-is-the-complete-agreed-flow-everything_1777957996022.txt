1. Super Admin Panel — What This Dev Builds
Audience: The platform owner (you / BookaWaka team). Not individual taxi companies.

Responsibilities:

Create and manage Companies (multi-tenant — each taxi company is a tenant)
Assign a Company ID to each tenant
Set which modules each company has access to (ACC module, Accounts module, Passenger App module, etc.)
Set subscription plans and billing for each company
Platform-wide settings: maps API, Firebase config, SMS gateway
Can view/support any company's data for support purposes
Does NOT touch: Individual jobs, drivers, ACC records, accounts — those belong to each company

Key rule for this dev: Every database table must have a company_id column. All queries filter by it. No company ever sees another company's data.

2. Owner / Manager Panel — What This Dev Builds
Audience: The taxi company owner or office manager (e.g. Invercargill Taxis Ltd)

Responsibilities (existing + to build):

Drivers & Vehicles (likely already exists)
Add/edit drivers, vehicles, zones, tariffs
Business Accounts tab (to build)
Create a corporate account: company name, contact, email, payment terms
System assigns an Account ID
Can deactivate accounts
Dispatcher searches this and attaches to jobs
ACC Clients tab (to build — the main new work)
ACC Client record (permanent — one per injured person):

Full name, phone number
Claim number ← permanent, never changes
Service code (PACS2, PACT1, etc.)
Wheelchair required (yes/no)
General notes
Purchase Orders — linked to that client:
Each time ACC sends a new approval letter/email, owner adds a new PO row:

PO number
Date from → Date to
Qty approved (total trips or trips per week)
Max price per trip (if specified)
Approved pickup address
Approved dropoff address
Manager name (Jan Legg, Barbara C, John Carter, etc.)
Manager email
Manager phone + extension
Branch / city (Invercargill, Hamilton, Dunedin, Taupo, Remote, etc.)
Status: Active / Expired / Upcoming (auto-calculated from dates)
Rules this dev must enforce:

Only one PO per client can be Active at a time (by date range)
When a renewal comes in — add new PO, old one auto-expires
Client address can change per renewal — store address on the PO, not the client record
Trips used auto-decrements when a job is completed against that PO
Reporting & Billing tab (to build)
End of month: select date range → system groups all completed ACC jobs by claim number → finds the active PO for each job date → generates invoice per PO
Invoice contains: client name, claim #, PO #, each trip (date / pickup / dropoff / amount), total
Send button → emails invoice to the manager email address stored on that PO
For Business Accounts: same flow but invoice goes to the company billing contact
3. Dispatcher Console — What This Dev Builds (that's me — current work)
Audience: The dispatcher on duty, watching the live board, creating jobs

How ACC and Accounts connect to the dispatcher:

The Customer Search box in the Create Job form searches three things at once:

ACC records → returns matching clients by claim number, name, or phone
Business Accounts → returns matching accounts by name, ID, or phone
Past passengers → returns anyone who has ridden before, by name, phone, or address
When dispatcher picks an ACC result:

Claim #, Remain (trips left), client name, phone fill in automatically — read-only
System finds the active PO for today — checks it's not expired, trips remain
If expired or no trips: error shown, cannot proceed
Approved addresses can optionally pre-fill pickup/dropoff (to build)
When dispatcher picks a Business Account:

Account ID, name, phone, email fill in — displayed so dispatcher can confirm
When dispatcher picks a Past Passenger:

Name, phone, email fill in — no claim, no account
Current known bug (to fix when ready):

Account ID, Claim #, Remain fields use value= not ng-model in AngularJS — display doesn't update reactively. Data saves correctly but dispatcher can't see it. Simple fix — swap to ng-model.
Dispatcher does NOT:

Create ACC clients or POs (that's Owner Panel)
Edit claim numbers or trip counts (read-only)
Manage accounts (that's Owner Panel)
4. Driver App — What This Dev Builds
Audience: The driver on their phone

Responsibilities:

Receive job offer (accept / reject)

Navigate (Google Maps hand-off)
Update job status: On Way → Arrived → Passenger On Board → Completed
If job is ACC-funded: driver sees it is ACC but does not see claim numbers or PO details
If job requires wheelchair: app flags it clearly on the job card
End of shift: summary of completed trips
Key connection:

When driver marks Completed → system decrements the trips remaining on the active PO (if ACC job)
Status updates fire back to dispatcher console in real time via Firebase
Does NOT handle: billing, account management, passenger registration

5. Passenger App & Website — What This Dev Builds
Audience: The end passenger booking their own ride

Responsibilities:

Register with name, phone, email, address
Book a ride: pickup, dropoff, now or later
Track driver on map
Pay by card (Stripe) or use credit
View booking history
If they are an ACC client: they book normally — the taxi company links their booking to the ACC claim on the back end (passenger doesn't manage their own claim)
Key connection:

When a passenger self-books, the job appears in the Dispatcher Console as an unassigned job — dispatcher (or auto-dispatch) picks it up
Passenger record is searchable from the dispatcher's Customer Search — so dispatcher can pre-fill their details on manual bookings too
Does NOT handle: ACC claims, corporate accounts — those are company-side records

What's Wrong or Missing
Problem	Detail
Wrong location	Creating ACC records belongs in the Owner Panel, not the Dispatcher Console. A dispatcher should only search and pick — not create or manage.
No Purchase Order concept	There's no distinction between the permanent Claim number and the PO number that changes on renewal. Right now it's all one record — so when a client renews you'd have no way to track history
Manager is too rigid	Currently a manager is created once and then picked from a list. But the same client can have a different manager on each renewal. Manager details belong on the PO record, not as a separate master entity
Missing fields from real ACC letters	No PO number, no max price per trip, no approved pickup address, no approved dropoff address, no wheelchair flag, no trips-used counter
Branch code not city name	Has "branch code" — your real letters show city names (Invercargill, Hamilton, Dunedin, Taupo, Remote)
No renewal tracking	No way to add a new period for the same client when their PO renews — would create a duplicate client each time
No trips used counter	Qty approved is stored but there's no running total of trips used vs remaining
No billing output	No connection to end-of-month invoice generation
