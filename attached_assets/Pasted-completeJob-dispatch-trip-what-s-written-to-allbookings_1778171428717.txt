job.paymentType is set at offer-accept time, not at completion time. The code (line 1732):

parsePaymentType (line 212) maps strings:

'card' / 'credit' → 'card'
'eftpos' → 'eftpos'
'account' / 'acct' → 'account'
'total_mobility' / 'tm' / anything with 'mobility' → 'total_mobility'
anything missing/unknown/undefined → 'cash' ← default fallback
Conclusion for a card-online booking: the driver app does not independently know the passenger paid online by card. It reads whatever PaymentType field dispatch included in the notification or allbookings at offer time. If dispatch didn't write PaymentType: 'card' into that payload, job.paymentType is undefined, parsePaymentType(undefined) returns 'cash', and at completion the driver app writes:

Summary for the dispatch team
There is a race condition / overwrite risk:

Passenger pays online by card → Stripe confirms → SA portal writes paymentMethod: 'card' to allbookings
Dispatch sends the job offer — if PaymentType: 'card' is included in the offer notification → driver stores it → completion writes paymentMethod: 'card' ✅
If dispatch does not include PaymentType in the notification → driver stores 'cash' → completion update() call overwrites paymentMethod: 'cash' back onto allbookings, clobbering whatever SA portal previously wrote ❌
So the stripeConfirmedFare guard alone is not sufficient if the driver completion write runs after the Stripe confirmation write — the driver will still overwrite paymentMethod to 'cash'. The fix needs to either:

Have dispatch always include PaymentType: 'card' in the notification payload for online-card bookings, or
Have the SA portal's sync endpoint re-assert paymentMethod after the driver app's completion write lands, or
Add a guard in completeJob to skip writing paymentMethod when the existing allbookings value is already 'card' (read-before-write, adds a round trip)
