GPS path confirmed and fixed. Your map listener was reading the wrong sub-node.

What the code actually does:

The dispatch map listener is on online/{cid}/{vid} — correct path, correct company isolation. No path change needed there.

The problem was one level deeper. When the driver app writes GPS to online/{cid}/{vid}/current, the child_changed event fires with the full node. The map code was then reading childsnapshot.val().lat — which is undefined because lat/lng live under current, not at the top level.

Fix deployed:

// Before the LatLng call:
var _gpsLat = driverData.lat || (driverData.current && driverData.current.lat) || 0;
var _gpsLng = driverData.lng || (driverData.current && driverData.current.lng) || 0;

Both the map marker position (google.maps.LatLng) and the distance-moved check (which decides whether to redraw the car) now use _gpsLat/_gpsLng. Top-level lat/lng is tried first so older driver app versions that write directly to the node still work.

Before this fix: every driver GPS update fired the child_changed event (so Firebase was working), but the map parsed parseFloat(undefined) = NaN and the car marker stayed frozen at (0, 0) or its last known position. All drivers appeared stationary.

After this fix: car markers move in real time as the driver app writes to online/{cid}/{vid}/current.
